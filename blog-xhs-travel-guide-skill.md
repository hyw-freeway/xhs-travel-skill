# 从「抄小红书攻略」到「一键生成可溯源攻略」：xhs-travel-guide Skill 的构建思考

## 一、项目背景：为什么做这个 Skill？

### 1.1 小红书成了旅行决策的「第一现场」

很多人做自由行前会去小红书搜「XX 几日游攻略」「XX 避坑」——真实用户的行程、踩坑和餐厅推荐，往往比官方介绍更贴近出行需求。但问题也明显：

- **信息分散**：要自己翻很多篇笔记，才能拼出完整行程和避坑清单。
- **质量参差**：标题党、软广和真实攻略混在一起，很难快速筛出高价值内容。
- **难以溯源**：收藏了一堆链接，写攻略时又懒得一条条贴回去，读者看不到「这话是谁说的」。

我们想做的是：**用 AI + 结构化流程，把「搜—筛—提炼—排版」变成一条指令**，并**强制每条建议都带上小红书原文链接**，让生成的攻略既可读又可查。

### 1.2 放在什么技术栈里做？

小红书没有官方开放 API，民间能力主要来自 [xiaohongshu-mcp](https://github.com/xpzouying/xiaohongshu-mcp)：用 Go 实现、通过 MCP（Model Context Protocol）暴露搜索、笔记详情、登录、发帖等能力。  
我们的场景是「只读聚合」：只用到搜索和拉取笔记详情，不发文、不点赞，对合规和账号安全更友好。

在此基础上，我们选择用 **Agent Skills** 来封装「生成旅游攻略」这一件事：把「何时触发、搜什么、怎么提炼、输出长什么样」写进一份 `SKILL.md`，让任意支持 Agent Skills 的客户端（如 Cursor、Claude Code、OpenClaw）都能用同一套逻辑工作，而不必每次重新写 prompt。

于是有了 **xhs-travel-guide**：一个输入「目的地 + 天数」（可选加「带父母」「不自驾」等偏好），输出「带小红书溯源链接的结构化 Markdown 攻略」的 Skill。

## 二、设计思路：可信、可执行、可发布

### 2.1 目标拆解

我们定了三个原则：

1. **可信**  
   攻略里每一句关键信息（景点、餐厅、避坑点、交通）都必须来自真实小红书笔记，并在正文中挂上可点击的链接；禁止「看起来合理但无来源」的幻觉内容。

2. **可执行**  
   读者能按攻略行动：有具体景点名、时段建议、预约渠道说明；交通和住宿要写清「怎么去、住哪一带」，而不是泛泛而谈。

3. **可发布**  
   输出是一份完整的、结构固定的 Markdown，用户可以直接复制到小红书后台，或稍作润色、换上自己的照片后发布，无需再大改版式。

### 2.2 多维度搜索：别只搜「X 日游攻略」

如果只搜「深圳三日游攻略」，很容易拿到一堆同质化的「Day1 去哪 Day2 去哪」，而**避坑、交通、门票预约**往往在别的笔记里。所以我们把「攻略」拆成四个信息维度，分别设计搜索词和排序策略：

| 维度       | 搜索词示例                         | 排序     | 目的                         |
|------------|------------------------------------|----------|------------------------------|
| 主干行程   | `{目的地} {天数}日游 攻略`         | 综合     | 整体路线与日程骨架           |
| 避坑/雷点  | `{目的地} 旅游 避坑` / `劝退`     | 最多收藏 | 真实负面体验与注意事项       |
| 交通出行   | `{目的地} 交通 攻略` 等           | 综合     | 市内/城际交通方式            |
| 门票/预约  | `{目的地} 景点 门票 预约`         | 最新     | 预约渠道与抢票提示（易过期）  |

这样 Agent 会在多轮搜索中覆盖「玩什么、别踩什么雷、怎么去、怎么约」，再在后续步骤里把信息合并进同一篇攻略。

### 2.3 溯源链接：每条建议都能点回原文

「有来源」不能只停留在「我读过小红书」——必须在正文里显式挂链接。我们做了几件事：

- **链接格式统一**：只使用 `https://www.xiaohongshu.com/explore/{feed_id}`，不夹带 `xsec_token` 等敏感或易过期参数。
- **相关性约束**：「当日参考攻略」挂到 Day X 的笔记必须与**当日行程内容相关**（同区域/同主题/同景点）；先根据笔记标题与正文判断是否与该日安排匹配，不匹配的不挂在该日。若多篇笔记均匹配当日，可全部列出。
- **覆盖率约束**：每一天至少 1 条与当日内容匹配的链接；交通、避坑、门票三个模块各自至少 1 条参考链接。这样读者在任意一节都能「点进去看原文」。
- **文末汇总表**：用表格列出所有被引用的笔记（标题、作者、链接、核心贡献），既方便读者按主题跳转，也便于作者自己核对来源。

### 2.4 只读、不写：边界清晰更安全

这个 Skill **只调用** `search_feeds` 和 `get_feed_detail`，不调用任何写接口（发布、点赞、收藏、评论）。这样带来的好处是：

- **合规与风控**：不替用户做任何账号行为，只做「读公开内容 + 整理」。
- **心智简单**：用户知道「只是查攻略」，不会误以为 AI 在代发笔记。
- **实现简单**：不需要处理发帖失败、审核、限流等问题，Skill 只关心「搜到什么、怎么写成 Markdown」。

同时在输出中明确注明「内容仅供参考」「来源为小红书公开笔记」，避免被当成官方或权威声明。

## 三、核心依赖：站在谁的肩膀上

### 3.1 xiaohongshu-mcp（能力层）

[xiaohongshu-mcp](https://github.com/xpzouying/xiaohongshu-mcp) 提供小红书侧的真实能力：

- **登录与会话**：扫码登录、Cookie 管理、登录状态检查。
- **搜索**：`search_feeds`，支持关键词、排序（综合/最新/最多点赞等）、内容类型与时间等筛选。
- **内容读取**：`get_feed_detail`，拉取单篇笔记的正文、图片、评论等。

xhs-travel-guide 只依赖「搜索 + 详情」这两类只读接口，不依赖发帖、互动等写接口。部署上，用户需要在本机或服务器先跑起 xiaohongshu-mcp 服务，再在 Cursor / Claude Code 等客户端配置 MCP 连接到该服务。

### 3.2 Agent Skills 规范（编排层）

我们采用 [Agent Skills](https://agentskills.io) 的约定，用 **单文件 `SKILL.md`** 描述一个能力：

- **name / description / argument-hint**：让宿主知道「这个 Skill 叫什么、干什么、参数长什么样」。
- **输入约定**：必填（目的地、天数）、可选（偏好），以及「缺信息时先追问」的规则。
- **执行流程**：分步骤写清「先确认需求 → 再按矩阵搜索 → 再拉详情提取 → 再合成 Markdown → 再输出文件」，这样大模型按文档执行即可，无需临时发挥。
- **输出格式**：固定五个模块（行程总览、交通与住宿、每日行程、门票与避坑、参考笔记表），保证每次生成的文档结构一致，便于自动化与二次加工。
- **约束与失败处理**：例如「只读不写」「未登录时引导用 xhs-login」「搜索无结果时建议换词或改天数」。

这样，Skill 的「契约」都在一份 Markdown 里，可版本管理、可复用在支持 Agent Skills 的各类客户端上。

### 3.3 MCP（协议层）

xiaohongshu-mcp 通过 **MCP（Model Context Protocol）** 把工具暴露给 AI 客户端。客户端（如 Cursor）配置好 MCP 服务地址后，就能在对话中调用 `search_feeds`、`get_feed_detail` 等。  
xhs-travel-guide 不直接实现 MCP，而是「约定好要调哪些 MCP 工具、怎么用返回结果」，具体调用由宿主模型根据 SKILL.md 的流程去完成。

## 四、Skill 构建过程的几点思考

### 4.1 用「搜索矩阵」代替「一次大搜索」

最初会想：搜一次「深圳三日游 攻略 避坑 交通」不就行了？实践下来发现：

- 小红书搜索对长 query 的意图拆分并不理想，混在一起往往拿到的还是清一色的行程帖。
- 避坑、交通、门票这类信息，在不同笔记里标题和关键词差异大，**分开搜、用不同排序**（例如避坑按「最多收藏」、门票按「最新」）更容易拿到高质量结果。

所以我们在 SKILL 里明确写了「四个维度、四类搜索词、各自排序建议」，把「搜什么」变成可复用的策略，而不是每次靠模型临时发挥。

### 4.2 输出结构固定：降低随机性，方便下游

如果让模型「自由写一篇攻略」，每次结构可能都不一样，后续要自动转 PDF、自动发小红书、自动提取「第几天」都会很麻烦。所以我们**强制**输出包含五个模块，且顺序固定：

1. 行程总览表  
2. 交通与住宿指南  
3. 每日详细行程（含当日参考链接）  
4. 门票预约与避坑指南  
5. 参考小红书笔记汇总表  

这样既方便人类「从上到下扫一眼」，也方便程序解析；同时约束了模型「必须填满这些块」，减少漏写或随意发挥。

### 4.3 「选帖策略」写进 Skill，而不是靠模型即兴

哪些笔记值得拉详情？我们在 SKILL 里写死策略：  
四维度搜索后**按 feed_id 去重**，总候选不超过 18 条；再按**互动量 + 标题关键词**（保姆级、血泪总结、亲测）排序，**最多**对 12 篇笔记调用 `get_feed_detail`。  
这样「质量过滤」和调用量都有上限，既控制 Token 消耗，又保证不同会话间行为一致。

**评论与 Token**：默认只用首屏约 10 条评论（`load_all_comments=false`）；仅对 2–3 篇与行程/避坑强相关的核心笔记才设 `load_all_comments=true` 且 `limit=15`，综合高赞评论中的补充信息。这样在保留「评论区金句」的同时，避免全量拉评论导致 Token 激增。

### 4.4 链接规范：可读性 + 隐私 + 长期有效

我们要求链接**只保留** `https://www.xiaohongshu.com/explore/{feed_id}`，不把 `xsec_token` 等写进正文。  
这样做的考虑是：  
- 链接短、干净，适合放在文章里和汇总表里；  
- 不暴露会过期的 token，减少「点开已失效」的体验；  
- 读者和平台都不会因为 URL 里带奇怪参数而产生顾虑。

同时要求「每一天 + 交通 + 避坑 + 门票」都至少有一条**与内容相关**的链接，且挂到某一天的链接必须与当日行程匹配，避免某一块完全变成「无来源总结」或「贴错日期的参考帖」。

### 4.5 高赞评论纳入综合

攻略不仅来自笔记正文，也来自**高赞/高回复的一级评论**（如「第二天建议早点去」「XX 店周一休息」）。在 SKILL 里约定：拉详情时默认带评论列表，但只有 2–3 篇核心笔记才加载更多评论并设上限（如 limit=15），其余用首屏约 10 条即可。这样既能把评论区里的实用信息写进攻略，又不会因全量拉评论而爆 Token。

### 4.6 失败与边界：未登录、无结果、缺参数

Skill 文档里单独写了「失败处理」：

- **未登录**：不自动发请求，而是引导用户先执行 `xhs-login` 完成扫码登录。  
- **搜索无结果**：建议用户放宽关键词、换目的地或天数，而不是硬凑一篇没依据的攻略。  
- **缺目的地/天数**：先追问补全再执行，避免跑了一轮发现「不知道在给谁写」。

这些约定让 Skill 在边界情况下行为可预期，也减少误用和无效调用。

## 五、小结与仓库

**xhs-travel-guide** 把「小红书攻略聚合」做成一个可复用的 Agent Skill：  

- **背景**：小红书已是很多人做旅行功课的主场，但信息分散、难溯源，我们想用 AI + 固定流程生成「可查来源」的攻略。  
- **思路**：多维度搜索（行程 / 避坑 / 交通 / 门票）、合并去重后最多 12 篇拉详情、综合高赞评论、强制溯源链接且「当日参考攻略」与当日内容匹配、固定输出结构、只读不写。  
- **依赖**：xiaohongshu-mcp（能力）、Agent Skills（编排）、MCP（协议）。  
- **构建思考**：用搜索矩阵代替一次大搜索、固定五段式输出、选帖去重与数量上限（含评论分级）控制 Token、参考帖与当日行程匹配、显式处理未登录与无结果等失败情况。

如果你在用 Cursor、Claude Code 或其它支持 Agent Skills 的客户端，并已部署 xiaohongshu-mcp，可以直接克隆本仓库，把 `.claude/skills` 拷到项目里，用 `/xhs-travel-guide 目的地 天数` 生成一篇带小红书链接的攻略试试。
